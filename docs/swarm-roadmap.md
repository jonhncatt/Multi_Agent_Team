# Swarm 路线图

## 当前状态

当前架构仍然是一个由后端编排的多 Agent 流水线：

1. 后端编排器接收用户输入
2. `Planner` 生成目标、约束和执行计划
3. `Worker` 执行主任务与工具循环
4. `Reviewer` 检查覆盖度、风险和置信度
5. `Revision` 在需要时改写最终答复

这还不是完整意义上的 swarm。它本质上仍然是一个固定顺序的串行流水线，只是已经具备较强的可观测性。

## 为什么现在还不算 Swarm

- 每一轮请求的角色是固定的
- 执行顺序写死在 Python 代码里
- 还没有 `Router / Coordinator` 角色
- 没有并行子任务
- 没有动态拉起多个同类 Worker 实例
- 没有多个并发 Agent 结果的聚合层

## 演进方向

目标方向是一个“轻量、可观测、可控”的 swarm：

- 后端继续作为主编排器
- LLM 负责辅助路由与分支决策
- 子任务可以拆给多个角色实例
- UI 仍然保持清晰的执行流程和角色交换可视化

## 阶段 1：打稳当前流水线

状态：进行中 / 大部分已完成

目标：

- 保持 `Planner -> Worker -> Reviewer -> Revision`
- 明确每个角色的职责边界
- 让调试流以“后端编排器视角”清晰可读
- 让 SSE 透明层稳定、实时
- 让某个角色失败时不污染整轮请求

退出标准：

- 团队能够解释调试面板上的每一步
- 工具调用轨迹清楚且能归因到 `Worker`
- `Reviewer / Revision` 在简单任务上的行为可预测

## 阶段 2：加入路由 / 分诊

状态：下一阶段

新增一个轻量角色：

- `Router` 或 `Coordinator`

职责：

- 判断任务类型
- 估计复杂度
- 决定是否需要 `Planner`
- 决定是否需要 `Reviewer`
- 决定是否需要 `Revision`
- 决定是否需要额外的专门角色

示例结构化输出：

```json
{
  "task_type": "simple_qa",
  "complexity": "low",
  "use_planner": false,
  "use_worker_tools": false,
  "use_reviewer": false,
  "use_revision": false,
  "specialists": []
}
```

建议规则：

- 简单知识问答：只走 `Worker`
- 中等复杂请求：`Planner -> Worker`
- 工具密集型请求：`Planner -> Worker -> Reviewer`
- 高风险 / 长文本 / 面向客户的请求：`Planner -> Worker -> Reviewer -> Revision`

这一阶段的重要性：

- 避免所有请求都固定跑 4 个角色
- 让编排开始体现“智能路由”
- 这是从流水线迈向 swarm 的第一步

## 阶段 3：专门角色

引入可复用的角色模板，例如：

- `Researcher`
- `FileReader`
- `Coder`
- `Tester`
- `Fixer`
- `Summarizer`

重要说明：

- 这不是动态发明新的 Python 类
- 而是预定义好的角色模板或 prompt
- 系统在运行时动态选择要实例化哪些角色

示例：

- `Researcher` 负责收集网页证据
- `Worker` 负责写答案
- `Reviewer` 负责检查答案和证据是否一致

## 阶段 4：多实例执行

允许同一种角色同时启动多个实例：

- `worker_1` 读取附件 A
- `worker_2` 读取附件 B
- `researcher_1` 搜索来源 A
- `researcher_2` 搜索来源 B

从这一阶段开始，系统才会明显呈现出 swarm 的感觉。

后端需要具备的能力：

- 子任务 ID
- 父子任务关系
- 共享 scratchpad 或聚合上下文
- 结果合并机制
- 针对子任务的超时和重试策略

## 阶段 5：并行分支

允许独立子任务并行执行。

示例：

- 同时读取附件和搜索网页
- 同时做代码分析和测试分析
- 并行收集多个来源摘要，再统一聚合

要求：

- 并发数可控
- 合并规则确定
- UI 能清楚展示并行分支
- 有成本和时延预算

## 阶段 6：聚合层

加入明确的聚合角色或聚合逻辑：

- `Coordinator` 收集多个子 Agent 的结果
- 解决结论冲突
- 选择最终答案的组织方式
- 决定是否需要再做一次验证

到了这一阶段，系统才会从“固定流水线”真正变成“可实战的 swarm”。

## 护栏

即使逐步演进到 swarm，也要保留这些护栏：

- 后端仍然是权限和工具策略的唯一真实来源
- 不是每个角色都能访问所有工具
- 专门角色必须职责收敛，避免变成万能 Agent
- 每一个被拉起的角色都必须在 UI 中可见
- 每一次结果合并都必须可检查
- 失败时要能优雅退化到更简单的路径

## 近期建议路线

1. 完成阶段 1 的 UI 和术语清理
2. 实现阶段 2 的 `Router`
3. 先只加一个专门角色：`Researcher` 或 `Fixer`
4. 等路由稳定后再做多实例支持

## 当前非目标

- 完全自治、开放式的 swarm
- 会自我修改代码的 Agent 系统
- 无限制动态创建角色
- 无法检查内部过程的黑盒编排
